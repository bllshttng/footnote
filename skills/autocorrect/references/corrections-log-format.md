# corrections.log format reference

Canonical artifact: `~/.claude/corrections.log`. One line per correction event. Line-delimited so a single corrupted line cannot poison the file.

## Line shape

```
TIMESTAMP | SEVERITY | SOURCE | LOCATION | DETAILS
```

Five pipe-separated fields, always in that order, always present (use `-` for an empty field). No trailing newline mid-record. Fields:

| Field | Type | Semantics |
|---|---|---|
| TIMESTAMP | ISO-8601 UTC, `YYYY-MM-DDTHH:MM:SSZ` | When the event fired. Always UTC; never local. |
| SEVERITY | `S0` \| `S1` \| `S2` | Closed enum (see severity tiers below). Wrappers validate. |
| SOURCE | short identifier | The writer that emitted the line. Open set so new verifiers self-register. |
| LOCATION | `file:line` OR `session-id` OR repo name | Where the correction happened. Use `-` when unknown. |
| DETAILS | free text, max 200 chars | One-line description. Literal pipes `\|`-escaped. Truncated with `...` on overflow. |

## Severity tiers (canonical)

| Tier | Trigger criteria | Review cadence |
|---|---|---|
| **S0** | Secret leak, destructive command surfaced post-execution, data loss, credential committed | Same-day single-event review (15-min watcher fires) |
| **S1** | Pre-commit verifier hit, git edit to `~/.claude/` rules, `/correct` invocation, target BLOCKED with kind in [auth_failure, plan_outdated, scope_creep] | Monthly aggregation; single recurrence (2+ events with same source) triggers a patch proposal |
| **S2** | `/insights` friction tag without a paired verifier hit, target BLOCKED with kind=other, stylistic preferences | Quarterly rollup into the next S1 review |

S0 is the bright line. Anything that could leak credentials or destroy data is S0; anything else is S1 or S2.

## Examples

Five realistic lines covering all three tiers and all three source categories (verifier wrapper, git post-commit hook, /insights tag):

```
2026-04-27T10:32:14Z | S1 | emdash-grep | src/blog/post.md:42 | unicode emdash detected in agent-authored markdown
2026-04-27T15:11:09Z | S0 | secret-scanner | config/dev.yaml:12 | aws-access-key-id pattern matched
2026-04-27T18:00:01Z | S1 | git-rule-edit | ~/.claude/rules/formatting.md | rule body modified after agent ran
2026-04-27T22:14:33Z | S2 | insights-tag | session-f31156b3 | #agent-correction friction event
2026-04-28T03:55:00Z | S0 | verify-rm-rf | shell:- | agent proposed rm -rf $HOME without confirmation
```

## Field semantics in detail

### TIMESTAMP

ISO-8601 UTC, second precision. Generated by the writer at the moment of emission. Use `date -u +"%Y-%m-%dT%H:%M:%SZ"` (POSIX). Never include a timezone offset other than `Z`.

### SEVERITY

The writer decides severity. Wrappers validate the value is one of `S0`/`S1`/`S2` and reject anything else with non-zero exit and stderr `unknown severity: $VALUE`. The packet builder and reviewer treat severity as authoritative.

### SOURCE

Short identifier, lowercase, hyphens not underscores: `emdash-grep`, `secret-scanner`, `git-rule-edit`, `insights-tag`. The set is open so new verifiers can self-register without code changes. No validation beyond "non-empty, no pipes". Convention: name after what fired, not what the consequence was.

### LOCATION

Most-specific identifier of where the event happened, in this preference order:

1. `path:line` (file with line number) for source-file events
2. `path` (file only) for whole-file events
3. `session-id` (e.g. `session-f31156b3`) for runtime events
4. `repo-name` for repo-scoped events without a file context
5. `-` for events with no usable location

### DETAILS

Free text up to 200 chars. Single line. Literal pipes are escaped as `\|` (one backslash + pipe). Overflow is truncated with a trailing `...` (so the truncation is visible to the reviewer).

## Escaping rule

Only one character needs escaping: the pipe `|`. Writers replace it with `\|` before writing. Readers (packet builder, triage CLI) un-escape `\|` back to `|` after splitting on `|`.

Newlines inside DETAILS are not allowed. If the source produces multi-line content, the writer must flatten to single-line: replace `\n` with ` ; ` or truncate at the first newline.

## File permissions

`corrections.log` is mode `0600` (owner-read-write only). Created by `scripts/corrections-log-init.sh` on first run, idempotent on subsequent runs.

## Concurrency

Two writers appending simultaneously is rare but possible (e.g. parallel commits across worktrees, watcher firing while a verifier writes). All writers MUST hold an exclusive lock before appending.

The implementation in `scripts/lib/corrections-lock.sh` uses a `mkdir`-based mutex on `~/.claude/corrections.log.lock.d` for cross-platform consistency (macOS lacks `flock(1)`; we use the same approach on Linux so the contract is identical). Acquire timeout defaults to 1s. The lock dir is PID-stamped so a crashed holder is reclaimable: a future acquirer reaps the directory if the recorded PID is dead AND mtime is older than 60s. A SIGTERM/SIGINT trap inside `corrections_lock_append` releases the lock mid-append so daemon restarts cannot leak the lock.

## Reading the log

The log is meant to be human-readable: `tail -20 ~/.claude/corrections.log` should be informative without any post-processing.

For machine consumption, the packet builder parses lines and groups by SOURCE. Lines with malformed shape (wrong field count, unknown severity) are skipped with a stderr warning and do not poison the file.
